Prompt para TUI / Cascade / SWE

Objetivo:
Corregir primero el problema de Tasks para que el estado "completed" sea real, verificable y visible. Prioridad máxima sobre cualquier otra mejora.

Reglas obligatorias:
- No teorizar.
- No explicar.
- No repetir contexto.
- No gastar tokens innecesariamente.
- No hacer cambios cosméticos.
- No tocar partes no relacionadas.
- Auditar primero el flujo real antes de cambiar.
- Hacer solo cambios concretos, verificables y mínimos.
- Al final entregar reporte corto, técnico y directo.

Tarea principal:
Auditar y corregir el flujo completo de Task detail + Task execution status para eliminar falsos "completed" y páginas en blanco.

Problemas a resolver:
1. /tasks/:id a veces carga fondo vacío y no muestra contenido.
2. Una task puede marcarse como completed aunque no sea verificable si realmente:
   - se intentó ejecutar
   - llegó al backend
   - llegó a OpenClaw
   - devolvió respuesta real
3. Al crear una task y encontrar agente compatible, el modal se queda abierto demasiado tiempo y no redirige correctamente a una vista comprobable.
4. Aunque la task termine, no hay confirmación visible y verificable del resultado real.

Trabajo requerido:

## FASE 1 — AUDITORÍA REAL DEL FLUJO
Audita estos puntos y úsalo como base del fix:
- creación de task
- creación de job
- dispatch del job
- ejecución real
- persistencia de generation trace
- transición a completed
- carga de /tasks/:id
- carga de paneles relacionados (timeline, diagnostics, generation trace, output)

Determina exactamente:
- qué condición está marcando completed
- si completed puede ocurrir sin respuesta real
- si completed puede ocurrir con fallback/stub
- si completed puede ocurrir sin confirmación de ejecución en OpenClaw
- qué endpoint o componente está rompiendo la vista de Task detail

## FASE 2 — FIX P0: TASK DETAIL NO PUEDE QUEDAR EN BLANCO
Implementar fix para que /tasks/:id nunca quede completamente en blanco.

Requisitos:
- añadir manejo de error por página y por panel
- si falla un panel, la página debe seguir mostrando el resto
- mostrar mensaje técnico mínimo en vez de fondo vacío
- no permitir que un error en diagnostics/timeline/generation trace rompa todo el render

Objetivo:
Task detail siempre visible aunque falte algún bloque.

## FASE 3 — FIX P0: "COMPLETED" DEBE SER REAL
Revisar y corregir la lógica de completado.

No marcar task como completed si no existe evidencia verificable de ejecución final.

Definir claramente qué evidencia mínima exige completed.
Como mínimo revisar si existe combinación coherente de:
- job ejecutado
- resultado final persistido
- generation trace o output final
- respuesta real o fallback explícitamente trazado

Si actualmente completed se asigna demasiado pronto:
- corregir la transición
- diferenciar claramente entre:
  - accepted
  - running
  - completed_verified
  - completed_with_fallback
  - failed
Si no es viable cambiar estados públicos completos, al menos:
- mantener completed solo cuando sea real
- exponer en UI si fue real response o fallback
- exponer si no hubo evidencia de ejecución OpenClaw

## FASE 4 — FIX P0: REDIRECCIÓN POST-CREACIÓN
Cuando se crea una task con éxito:
- cerrar modal al terminar creación
- redirigir a /tasks/:id
- mostrar estado visible de creación/asignación/ejecución
- permitir comprobar rápidamente si está running, completed o failed

No dejar al usuario esperando en el modal sin visibilidad.

## FASE 5 — VERIFICABILIDAD EN TASK DETAIL
En la vista de task completar al menos estos bloques visibles:
- status real
- agent asignado
- job asociado
- execution mode final
- fallback usado o no
- output final o ausencia explícita
- generation trace disponible o no
- marca clara si hubo ejecución real vs fallback/stub

No asumir nada.
Si falta evidencia, mostrarlo explícitamente.

## FASE 6 — VALIDACIÓN
Probar al menos estos escenarios:
1. task con agente compatible y respuesta real
2. task con fallback
3. task que falla
4. task rápida que termina antes de abrir detalle
5. task detail con un panel roto: la página no debe quedar vacía

## REPORTE FINAL
Entregar reporte breve con:
- causa real del falso completed
- causa real de la página en blanco
- archivos tocados
- flujo corregido
- qué evidencia usa ahora completed
- qué sigue pendiente

Restricciones:
- no abrir refactor grande
- no tocar herramientas/skills/agentes todavía
- no mezclar con otros bugs
- enfocarse solo en Task detail + truthfulness de completed + redirección post-creación
